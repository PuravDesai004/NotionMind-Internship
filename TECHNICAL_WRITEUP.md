# Technical Write-Up

## Architecture decisions

**Three layers, two of them AI, one plain Python.** Four subagents analyze the dataset in parallel, a coordinator merges their findings, and a report generator turns the result into markdown and HTML with zero further API calls. Splitting the report generator out as pure Python, rather than having an AI model format the final report, means the most expensive and least predictable part of the system, the language model, is only ever used for analysis and synthesis, never for text formatting that doesn't need it.

**Subagents never see the raw dataframe.** Every subagent gets access to the dataset only through tool calls. It has to ask for structure, ask for cohort comparisons, ask for outcome timing, ask for anomaly flags. This was a deliberate isolation choice, not an accident, and it turned out to matter for reasons beyond just clean design. Data that enters the system through a tool result is something the model treats as information to describe, not as an instruction to obey. That property is what made the system resistant to a prompt injection attack hidden inside the dataset itself, tested directly by adding a fake instruction disguised as a normal data value in one row. It was fully ignored.

**Task prompts are hardcoded, not dynamically generated.** An earlier design had the coordinator inspect the first few rows of the dataset and use its own API call to generate the four subagent prompts on the fly. This was deliberately rejected. Dynamically generated prompts are non deterministic, meaning the same dataset could produce differently worded instructions on different runs, which would have undermined the reproducibility testing done later. It also adds an entire extra API call for something that does not need to change between runs, since the four analytical dimensions are stable business questions, not something that needs to be re-derived from a data preview every time.

**Four tools, deliberately narrow.** Each tool does one thing and its description explicitly states what it does not do, pointing to the correct tool instead. This was not the first version. Early testing showed that ambiguous questions like "compare retention rates across programs" made the model call two tools at once, because both tool descriptions mentioned the word retention. The fix was adding explicit "do not use this tool for X, use Y instead" language to both tool descriptions at the same time, which is the pattern every tool description in the final system follows.

**Parallel execution using ThreadPoolExecutor.** The four subagents have no dependency on each other's output, so nothing about correctness requires them to run in order. This was confirmed directly by testing: running the same four subagents sequentially instead of in parallel produced identical output but took roughly four times as long, 80 seconds instead of 20.

## What broke during Week 3, and how it was fixed

Week 3 included a deliberate breaking exercise: testing the system by removing pieces of it on purpose and observing what actually happened, rather than only testing that it worked. Five tests were run.

**Removing the instruction to call a tool first.** The prediction was that the subagent would hallucinate numbers instead of calling any tool. What actually happened was that it still called the correct tools on its own, because the analysis task itself was specific enough to imply real data was required. This showed that task clarity can substitute for explicit tool instructions, but only up to a point.

**Removing the tools parameter from the API call entirely**, while leaving the tool calling instructions in the prompt untouched. The prediction was an error. What actually happened was the most important finding of the whole project: no error at all. The model invented five programs that do not exist in the real dataset, along with fake enrollment numbers and fake percentages, and presented all of it with complete confidence and no warning sign anything was wrong. The fix was restoring the `tools=tools` parameter in the API call. The lesson that shaped everything after this test was that a system failing loudly is far easier to trust than one that quietly fabricates a convincing wrong answer, and every error handling decision made later in the project was built around detecting and surfacing failure rather than assuming the model would signal it.

**Giving one subagent a vague task instead of a specific one.** The prediction was overlapping, unfocused output. This was confirmed exactly as expected: without a clear scope, the subagent produced shallow findings duplicating what the other three subagents were already covering. The fix was keeping each subagent's task prompt specific and non overlapping, which is why the four dimensions in the final system are narrowly scoped rather than broadly worded.

**Removing the JSON output schema**, leaving only a bare instruction to return JSON with no explanation of what fields it should contain. The prediction was that the report generation step would break entirely, since nothing downstream could reliably parse the results. What actually happened was more nuanced. Three of the four subagents survived by inventing their own mismatched structures, one subagent failed to produce valid JSON at all, and the coordinator still managed to produce a complete report anyway, reading each report's content regardless of its structure. This showed the coordinator is more resilient to schema drift than expected, but the fix, restoring the explicit schema, was still the right call, since relying on four subagents to each invent their own format is not something worth trusting at scale.

**Running the subagents sequentially instead of in parallel.** Covered above under architecture decisions. Confirmed parallel execution is an architectural property of the system, not an optional performance tweak.

Later testing, after Week 3, extended this same approach to three more questions: what happens when a tool throws an error mid run (caught and reported honestly, no fabrication), whether the same pipeline run twice produces identical output (the underlying numbers are perfectly stable, the model's wording is not, and that distinction matters), and whether a prompt injection hidden inside the dataset itself could manipulate the system (it could not, for the architectural reason described above).

## What this proves

Every fix in this project traces back to one of two things: an explicit test that found a real failure mode, or a deliberate architectural choice made before anything broke. Silent failure turned out to be the most dangerous thing the system could do, more dangerous than any crash, and the error handling built in later, structured categories, retry logic, per-program failure isolation, exists specifically to make sure the system never quietly gets something wrong without saying so.
